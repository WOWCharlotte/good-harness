# 我把 Agent Memory 从入门到实践讲明白：别让 AI 明天就忘了今天的你

> 我这篇文章想解决一个问题：Agent Memory 到底是什么，工程上该怎么做，以及为什么它不是“把聊天记录存起来”这么简单。
> 
> 我会尽量用第一人称把逻辑讲顺：该专业的时候专业，该吐槽的时候吐槽。毕竟，AI 可以没有情绪，但我写文章不能像日志服务器。

---

## 0. 前言：我先讲一个差点把 AI 带沟里的真实故事

2026 年 4 月，Jina AI 创始人肖涵跟他的 AI 助手“小牛马”说：

> "帮我把之前做的那个 911 的图表发给我。"

AI 回复："没找到。你说的 911 是指恐袭吗？"

但肖涵说的 911，是保时捷 911 跑车。

他开始 debug。我看到这里也愣了一下：AI 用了正确的搜索工具（Grep + Memory Search + 向量语义搜索），跑的是当时最好的模型（Claude Opus 4.6），存储链路都没问题。

**问题偏偏出在第一步：Query 构造。也就是说，不是工具不会干活，而是开局就把路带歪了。**

- 第一次 Query：`911 chart graph visualization`（把中文翻成英文，丢失了上下文）
- 第二次更离谱：`September 11th chart visualization Twin Towers attack`（直接搜成恐袭了）

当他说"帮我把 992.2 的图表发给我"（992.2 是 911 这代车型的代号），一下就找到了。

**这个故事让我意识到：即使记忆系统整条链路都对，只要 Query 构造这一步出问题，整个检索还是会当场翻车。记忆，确实是今天 AI Agent 最大的痛点之一。**

## 1. 我理解的 Agent Memory：到底记的是什么？

![](images\1.png)

### 1.1 我为什么觉得 Agent 必须有记忆？

我还记得 2023 年刚开始用 LLM 的时候，AI 像一个"短期失忆症的天才"——它可以完美回答你当下的问题，但关掉对话窗口后，它就忘了你是谁。

举个直观例子：

> **你**：我不吃辣。
> **AI**：好的，我记住了。
>
> (三天后，新对话)
>
> **你**：推荐一家餐厅。
> **AI**：这家麻辣火锅很不错！

没有记忆，AI 每次都是"初次见面"。

这哪是智能，简直是电子版“初次见面，请多关照”。我只能说：血压上来了。

### 1.2 所以，Agent Memory 到底是什么？

**在我看来，Agent Memory 就是跨会话持续存在的、可影响未来决策的结构化历史。**

- **跨会话（cross-session）**：不只是在一次对话中记住上下文
- **结构化（structured）**：不是"把聊天记录再存一份"，而是带来源、作用域、时间权重和可修正性的历史对象
- **可影响决策**：存了要能用，要在未来决策中产生差异

### 1.3 我先拆几个容易混的概念

| 概念 | 本质区别 |
|---|---|
| **Memory vs RAG** | RAG 是静态"图书馆"（书不变），Memory 是动态"大脑"（内容持续更新、演化） |
| **Memory vs Context Engineering** | Context Engineering 管"如何塞东西进有限窗口"，Memory 管"我是谁、我经历了什么" |
| **Memory vs State** | State 是当前 session 内的临时运行态（session 结束即销毁），Memory 是跨 session 持续存在的 |

### 1.4 我会从三个维度给记忆分类

#### 按内容性质分

- **情境记忆（Episodic Memory）**：记住"发生了什么"——具体的事件、经历、时间线
- **语义记忆（Semantic Memory）**：记住"是什么"——事实、规则、用户偏好
- **程序性记忆（Procedural Memory）**：记住"怎么做"——技能、流程、操作步骤

#### 按时间尺度分

- **短期记忆/工作记忆（Short-term/Working Memory）**：当前会话内的上下文，受 context window 限制
- **长期记忆（Long-term Memory）**：跨会话持久化的信息，存储在外部系统中

#### 按存储形式分

- **参数化记忆（Parametric Memory）**：通过模型权重存储（微调/预训练写入的知识）
- **非参数化记忆（Non-parametric Memory）**：通过外部存储系统（向量库、图数据库、文件系统）

## 2. 短期记忆与上下文工程：别让窗口一满就“大脑宕机”



### 2.1 短期记忆的问题是什么？

每次你和 AI 交互，AI 看到的其实是一个"上下文窗口"（Context Window）里的内容。这个窗口有限制——比如 128K tokens，当对话变长、工具输出增多、文档内容塞满，窗口就会溢出。

**短期记忆管理的本质**：在有限 context window 下，保持推理所需的关键信息不丢失。

### 2.2 我常用的三种核心策略

#### 策略一：上下文压缩（Context Reduction）

减少信息量，降低 token 消耗：
- **保留预览**：大块内容只保留前 N 个字符
- **总结摘要**：用 LLM 对长内容做摘要，保留关键信息

#### 策略二：上下文卸载（Context Offloading）

被压缩的内容存到外部（文件系统/数据库），消息中只保留引用（文件路径/UUID）。需要时按引用重新加载。

**优势**：信息不丢，上下文干净。

#### 策略三：上下文隔离（Context Isolation）

通过多智能体架构，把上下文拆分到不同子智能体中。主智能体只写任务指令，子智能体各自在自己的上下文中执行，返回结果。

**适用**：任务有清晰简短指令，只有最终输出才重要（如代码库搜索）。

## 3. 长期记忆：我把它拆成 Record 与 Retrieve

我会把核心流程理解为：长期记忆和短期记忆形成双向交互

- **Record（写入）**：从短期记忆的会话消息中，通过 LLM 提取有效信息（事实、偏好、经验），写入长期记忆

- **Retrieve（检索）**：根据当前用户查询，从长期记忆中检索相关信息，注入到上下文中辅助推理

```
         ┌──────────────────────┐
         │    短期记忆/上下文     │
         │   (用户输入+模型回复)   │
         └──────────┬───────────┘
                    │
          Record ↓  │  ↑ Retrieve
          (写入)    │  │  (检索)
                    │  │
         ┌──────────┴───────────┐
         │    长期记忆存储        │
         │  (向量库+图数据库+...)  │
         └──────────────────────┘
```

**要让长期记忆真的跑起来，我认为至少需要以下组件：**

| 组件 | 作用 |
|---|---|
| **LLM** | 提取短期记忆中的有效信息（语义理解、抽取、决策） |
| **Embedder（向量化模型）** | 将文本转换为语义向量，支持相似性计算 |
| **VectorStore（向量数据库）** | 持久化存储记忆向量和元数据，支持高效语义检索 |
| **GraphStore（图数据库）** | 存储实体-关系知识图谱，支持复杂关系推理 |
| **Reranker（重排序器）** | 对初步检索结果按语义相关性重新排序 |
| **SQLite/日志系统** | 记录所有记忆操作的审计日志，支持版本回溯 |

> 读到这里你可能会发现：长期记忆和搭建一个 RAG 的流程很像。我的理解是，它确实可以看成一个“超级进化版 RAG”——RAG 穿上记忆盔甲，准备打副本了，嘻嘻嘻~

**长期记忆 vs RAG 的关键区别：**

| 维度 | RAG | Agent Memory |
|---|---|---|
| 数据来源 | 静态文档 | 动态对话+交互历史 |
| 更新方式 | 手动/批量更新 | 实时自动提取 |
| 存储内容 | 文档片段 | 事实/偏好/经验 |
| 更新粒度 | 全量替换 | 增量更新，冲突检测 |
| 个性化 | 无 | 用户级/会话级/代理级 |
| 时间感知 | 通常无 | 时间戳+衰减+演化 |

## 4. 记忆的本质：我认为核心是 Ledger + Views + Policy

> 这是我认为 2026 年最关键的架构洞见之一：Memory 不是一个孤立组件，而是一个闭环系统。



![](images\3.png)

我大概总结了目前一些比较好的 idea，并把它们压缩成下面三个命题：

**命题A：Memory不是"存储"，而是可被决策利用的外部状态。**

存了很多历史不等于有记忆能力。能力的来源是在当前状态下，历史能否以某种形式影响决策分布。

**命题B：Memory的最小闭包是 (Ledger, Views, Policy) 三件套。**

只要系统想满足可溯源、可回滚、可观测这些硬约束，Memory的最小形态就不可能只是"一个向量库 + 若干 prompt"。它必须是一个能被审计的状态机。

**命题C：Memory的基本单位是 event 序列，但event流本身不等于可用的记忆系统。**

event序列是"真相来源"，但它太底层。真正把历史变成能力的是 views（对event的重组织/压缩/索引）和 policy（决定什么时候触发、怎么触发）。

> 我把这套东西理解成：先有账本，再有索引，最后再有调度策略。没有账本的记忆，像没有聊天记录的群聊复盘——各说各的。

**Raw Ledger（权威记录）：**
- 追加式记录每次写入/更新/删除
- 记录当时的输入、时间、scope
- 像"账本/黑匣子"，支持溯源和回滚

**Derived Views（派生视图）：**
- 面向检索/推理的派生状态
- 包括：向量索引、关键词索引、知识图谱、时序知识图谱、timeline、skill index
- views 可以有多个、可以lossy，但必须可回指到 Raw Ledger
- 像"缓存 + 索引 + 物化视图"

**Policy（控制层）：**
- 决定何时读、读多少、何时写、如何更新、如何遗忘
- 决策必须显式化为 ADD/UPDATE/DELETE/NONE 等 Action 序列
- 像"调度器/控制回路"

> **三者缺一不可：没有Ledger，系统不可治理；没有Views，系统不可用；没有Policy，系统不可持续迭代。**

```
┌─────────────────────────────────────────────────┐
│                  Policy（控制层）                  │
│         决定何时读、读多少、何时写、              │
│         如何更新、如何遗忘                        │
│         ┌──────────┐     ┌──────────┐            │
│         │  Commit  │     │Provenance│            │
│         │  (回写)   │     │ (可回放)  │            │
│         └────┬─────┘     └────┬─────┘            │
├──────────────┼───────────────┼───────────────────┤
│              │               │                   │
│    ┌─────────┴───┐  ┌───────┴──────┐            │
│    │ Raw Ledger  │→│ Derived Views │            │
│    │  (权威记录)   │  │  (派生视图)    │            │
│    │ Append-only │  │ 向量/关键词/   │            │
│    │  事件日志    │  │ KG/TKG/Timeline│            │
│    └─────────────┘  └──────────────┘            │
└─────────────────────────────────────────────────┘
```

**如果记忆不是一句摘要，那它到底是什么？**

需要六个维度：

| 维度 | 说明 | 示例 |
|---|---|---|
| **内容（Content）** | 这条记忆说了什么 | "用户偏好 TypeScript" |
| **类型（Type）** | 至少五种：event / assertion / belief / constraint / commitment | belief（推断出来的）vs assertion（用户明确声明的）——更新机制不同 |
| **置信度（Confidence）** | 对这条记忆有多确信 | assertion 通常高置信，belief 需要通过新证据修正 |
| **来源（Provenance）** | 从哪来：用户表达 / 行为推断 / 环境观察 / Agent自生成 | 决定了可信层级和可撤销性 |
| **作用域（Scope）** | 在什么上下文下成立 | "偏好性能优先"在后端架构决策中成立，在前端原型阶段未必 |
| **时间与衰减（Time & Decay）** | 什么时候产生、上次确认/引用时间、衰减权重 | 去年的偏好不等于现在 |

> **六维度钉住，记忆就从"一堆字符串"变成了可查询、可追溯、可修正、可过期的结构化对象。**

## 5. 实际工程里，我会怎么做 Agent Memory？

### 5.1 多 Agent 框架里的记忆架构

![](images\4.png)

到了多智能体场景，记忆问题会直接升级成“群聊管理地狱”：**多个Agent之间如何共享信息？哪些记忆应该隔离？谁说过的、谁该记谁的、谁不该看谁的？**

#### 5.1.1 多 Agent 记忆的核心挑战

**记忆边界**：哪些记忆是Agent私有的，哪些是共享的？
**消息路由**：Agent A产生的信息，Agent B能不能看到？
**冲突处理**：两个Agent对同一个事实有不同认知怎么办？
**记忆膨胀**：多个Agent的交互量远超单Agent，如何防止记忆库爆炸？
**权限治理**：群聊场景中，Agent该记谁的、不记谁的？

#### 5.1.2 开源多智能体框架是怎么解决的？

##### 1. LangGraph

LangGraph 是 LangChain 出品的 Agent 编排框架，其记忆模型围绕 **State + Checkpointer** 构建：

- **State（状态）**：每个 Agent 节点通过共享的 State schema 进行通信。State 在 Graph 内的节点之间流转，相当于一个结构化的短期记忆/工作区
- **Checkpointer**：将 Graph 执行的中间 State 持久化到外部存储（SQLite/Postgres）。支持断点续跑、回溯到任意历史节点、跨会话恢复
- **长期记忆**：通过 `Store` API 实现跨会话的持久化记忆。每个 Agent 节点可以读写 Store，支持 namespace 隔离（不同用户/租户的记忆天然隔离）
- **记忆分层**：State（工作记忆/图内流转）→ Checkpointer（会话级持久化/断点续跑）→ Store（长期记忆/跨会话）。三层各司其职

##### 2. CrewAI

CrewAI 是多Agent编排领域Star数最高的框架，其记忆设计注重**开箱即用**；CrewAI 的记忆特点是**按 Crew（团队）维度组织**——同一个 Crew 中的 Agent 共享记忆池，不同 Crew 的记忆相互隔离。

- **短期记忆**：当前Crew执行期间的上下文，使用 OpenAI embedding + 向量存储
- **长期记忆**：跨Crew执行的任务结果和经验，持久化到本地SQLite
- **实体记忆**：对用户/实体的事实信息（偏好、属性等）
- **用户记忆**：与特定用户相关的个性化信息

##### 3. AutoGen（Microsoft）

AutoGen 的精妙之处在于：**不强制所有Agent共享一个全局记忆池**，而是让每个Agent各自维护记忆，通过对话历史自然地共享信息。

AutoGen 的记忆模型围绕 **AgentChat** 构建：

- **对话历史**：以消息列表的形式存储，是多Agent之间共享的基础记忆
- **Agent Memory**：每个 Agent 可以持有自己的记忆（通过外部向量存储）。Agent 在对话中按需检索自己的记忆库
- **Termination Condition**：通过终止条件来控制对话/记忆何时结束，避免无限膨胀
- **Handoff 模式**：Agent 间通过 handoff 传递控制权和上下文，确保对话流转时关键信息不丢失

##### 4. MetaGPT

MetaGPT 引入了类似软件工程中的 **Shared Message Pool（共享消息池）** 概念：

- **消息池（Environment）**：所有 Agent 发布消息到一个共享环境，其他 Agent 可以订阅自己关心的消息
- **角色记忆**：每个 Agent 通过 `role_id` 隔离自己的记忆，只需要关注与自己角色相关的历史消息
- **Short-term Memory**：当前任务的上下文和中间产出
- **Long-term Memory**：跨任务的经验和知识

##### 5. AgentScope

AgentScope 是国内多Agent框架的代表：

- **短期记忆**：`AutoContextMemory`，6种渐进式压缩策略，从轻量级到重量级
- **长期记忆**：内置 `ReMe` 组件，支持跨会话记忆；也支持集成第三方（如 Mem0）
- **记忆隔离**：通过 `Memory` 接口支持 Agent 级别的记忆隔离——每个 Agent 可挂载不同的 Memory 实例
- **Dist 模式**：在分布式部署下，不同Agent跑在不同进程中，各自持有独立记忆上下文

##### 6. MIRIX（2025.07）

MIRIX 是专门围绕"记忆分治"设计的Multi-Agent记忆系统，在 LOCOMO 评测上取得了 SOTA 成绩；

**MIRIX 的核心思想**：将记忆的管理从Agent的主任务中"外包"出去——让专门的子Agent负责记忆路由和调度，而不是让每个Agent自己管理全部记忆。

- **6个记忆组件**：将记忆按功能拆分为6个独立的模块，每个模块有独立的存储结构和检索方式
- **Meta Memory Manager（元记忆管理器）**：一个专门负责记忆路由的子Agent，分析输入该更新/检索哪些记忆组件，再将指令分别路由到对应的记忆管理器
- **层级检索**：第一轮从所有记忆组件做粗略检索，只返回高层次摘要；Chat Agent 根据摘要决定需要深入搜索哪些组件，第二轮才返回详细内容

我把上面这些框架压缩成一句话：多 Agent 记忆架构大概逃不出以下几个方向：

| 模式 | 说明 | 代表框架 |
|---|---|---|
| **共享消息池** | 所有Agent往同一个池子发布/订阅消息，池子本身即记忆 | MetaGPT |
| **共享State+持久化** | Agent通过共享State通信，Checkpointer/Snapshot持久化 | LangGraph |
| **团队隔离** | 按Crew/Team维度组织记忆，队内共享、队间隔离 | CrewAI |
| **Agent各自记忆** | 每个Agent独立维护记忆库，通过对话历史自然共享 | AutoGen |
| **记忆外包** | 专门子Agent负责记忆路由和调度，主Agent只消费结果 | MIRIX |
| **全局+局部双层** | 全局记忆池 + 每个Agent的私有记忆层 | AgentScope |

### 5.2 我会重点关注的开源记忆工具

#### Mem0 — 最通用的"即插即用"记忆引擎

https://github.com/mem0ai/mem0 （54.3K stars）

Mem0 是目前 Star 数很高的独立记忆工具，提供 Python SDK 和 REST API，几乎可以嵌入任何Agent框架；

多Agent场景下：
- **user_id + agent_id 双维度隔离**：同一用户在不同Agent间的记忆可以隔离，也可以跨Agent共享
- **category 标签**：按自定义目录分类记忆，不同Agent只检索自己关注的category
- **MCP协议支持**：Mem0 提供了 MCP Server（OpenMemory），Agent通过标准MCP协议调用记忆，无需在代码层集成
- **异步记忆生成**：记忆提取在后台异步完成，不阻塞Agent主流程——这对多Agent并发场景尤其重要

#### Graphiti — 时序知识图谱记忆

https://github.com/getzep/graphiti （25.5K stars）

Graphiti 是一个用于构建和查询 AI 代理时间上下文图的框架。

与静态知识图不同，Graphiti 的上下文图跟踪事实随时间的变化，保持对来源数据的来源，并支持两者 规定和学习的本体论——这也使它更适合服务那些需要处理不断变化的现实世界数据的智能体。

Graphiti 的核心是**时序知识图谱**，适合需要可解释性和时序推理的多Agent场景；

多Agent场景下的集成要点：
- **双时间轴（bi-temporal）**：记录"事件发生时间"和"记入图谱时间"，Agent可以回溯"什么时候知道了什么"
- **边失效机制**：新事实与旧事实冲突时，旧边不删除而是标记为"已失效"——多个Agent更改同一事实时保证可追溯
- **实体去重**：不同Agent提取的相似实体通过全文检索+LLM判重合并，防止多Agent产生重复实体
- **自定义实体/关系类型**：通过Pydantic模型定义领域实体，不同Agent可以使用不同的实体schema

#### Cognee — 利用知识引擎为agent构建个性化与动态记忆

https://www.zdoc.app/zh/topoteretes/cognee（16.9K stars）

Cognee 是一个开源的知识引擎，允许以任何格式或结构摄取数据，并持续学习，为 AI Agent 提供正确的上下文。它结合了向量搜索、图数据库和认知科学方法，使文档既能通过含义进行搜索，又能随着其变化和发展通过关系相互连接。

多Agent场景下的集成要点：
- **RDF/OWL 本体支持**：用标准本体语言定义知识结构，不同Agent/团队可以共享同一套本体标准
- **Pipelines（管道）**：Task → Pipeline 的可组合工作流，不同Agent可以走不同的抽取/检索管道
- **dataset_name 隔离**：按数据集维度隔离记忆，不同Agent挂载不同的数据集组合
- **多源数据互联**：支持30+数据类型（文本/图片/音频/PDF等），面对多源异构Agent输入天然占有

#### LangMem — LangChain生态的记忆SDK

https://github.com/langchain-ai/langmem (1.4K)

LangMem 是 LangChain 推出的专用记忆管理 SDK;

- **Memory Types**：语义记忆（facts about user）+ 程序记忆（procedures Agent learned）+ 情节记忆（episodic events）
- **extract()**：从对话中提取记忆，输出结构化数据
- **optimize()**：管理记忆质量——去重、冲突检测、遗忘
- **核心设计理念**："记忆是提示词管理问题，不是存储问题"——LangMem 不绑定特定存储后端，而是提供记忆操作逻辑

## 6. 我再看看别人的 Agent Memory 是怎么做的

我把它分成三大流派与三巨头

![](images\5.png)

> 区分所有记忆产品的第一性原则：**Source of Truth（真相源）到底在哪？**

### 6.1 三大流派

#### 6.1.1 数据库派

**真相源：结构化存储（向量库/SQL/图数据库）。**

对话经过 LLM 抽取 facts，存进向量库或图谱，检索时注入上下文。

| 代表 | 特点 |
|---|---|
| **Mem0** | 向量+图混合存储，自动冲突检测，最易落地 |
| **Graphiti** | 时序知识图谱，双时间轴，企业级场景 |
| **Cognee** | 向量+图谱+RDF/OWL本体，最灵活 |

**优势**：结构化程度高，查询效率好。
**劣势**："一条记忆长什么样"由 schema 锁死，灵活性有限。

#### 6.1.2 文件派

**真相源：Markdown/纯文本文件。**

Agent 读文件 → 工作 → 写回文件 → 不断累积。

| 代表 | 特点 |
|---|---|
| **Claude Code / Codex** | memory文件夹 + Markdown日记 + soul.md + memory.md |
| **OpenClaw** | MEMORY.md + 每日日志 + 可插拔Context Engine |

**优势**：透明可编辑、可版本化（git diff）。
**劣势**：文件会膨胀，需要 intelligent forgetting。

#### 6.1.3 模型派

**真相源：大模型权重或上下文本身。**

模型自决定记什么、忘什么、整合什么。代表：Letta、ChatGPT用户记忆、千问3.5（1M token上下文）。

**优势**：零配置、自适应，极限情况下不需要外部系统。
**劣势**：完全黑盒、不可审计、记忆不可跨模型迁移。

我看下来，2026 年的主流工作流已经形成了一个比较明显的共识：

```
对话记录
    │
    ▼
LLM 提取事实/记忆结构
    │
    ├──→ 向量化存储（语义检索）
    ├──→ 知识图谱存储（关系推理）
    ├──→ 时序数据库存储（时间感知）
    │
    ▼
混合检索（向量 + 关键词 + 图遍历）
    │
    ▼
冲突检测 + 时间衰减
    │
    ▼
注入上下文，辅助推理
```

### 6.2 2026 年 Agent 工具三巨头

OpenClaw、Hermes、Claude Code 是 2025-2026 年 Coding Agent 领域讨论度很高的三个工具。

| 维度 | **Claude Code** | **OpenClaw** | **Hermes**（开源产品） |
|---|---|---|---|
| **核心哲学** | 脚手架越轻越好——模型够聪明，给它看到就行 | 先建管道——基础设施不过期 | 自主进化——Agent 通过技能闭环持续成长 |
| **开源情况** | ❌ 闭源（Anthropic） | ✅ 开源（Peter Steinberger创建） | ✅ 开源（Nous Research，MIT） |
| **记忆哲学一句话** | "模型直接读文件，不需要复杂中间层" | "记忆做成可插拔的系统服务" | "多层级记忆 + 技能闭环 + 可插拔Provider" |
| **面对模型进步** | 做减法——组件可安全移除 | 建管道——基础设施可持续用 | 做生态——Provider可插拔、Skills跨平台复用 |

### 6.3 Claude Code 记忆设计：文件派标杆

Claude Code 是文件派记忆设计的代表。它的核心理念我概括成一句话：**“真相在文件里。”**

```
┌─────────────────────────────────────────────────┐
│                  Claude Code                     │
│                                                  │
│  真相源（Source of Truth）                        │
│  ┌──────────────────────────────────────────┐   │
│  │  memory/ 文件夹                            │   │
│  │  ├── 2026-04-28.md  (按天组织的Markdown日记) │   │
│  │  ├── 2026-04-27.md                        │   │
│  │  ├── 2026-04-26.md                        │   │
│  │  ├── memory.md      (长期记忆：用户画像)     │   │
│  │  └── soul.md        (自我画像：自身认知)     │   │
│  └──────────────────────────────────────────┘   │
│                    │                             │
│                    ▼                             │
│  派生索引层（从文件派生，不替代文件）              │
│  ┌──────────────────────────────────────────┐   │
│  │  SQLite FTS5  ──── BM25 关键词全文检索      │   │
│  │  LanceDB       ──── 向量语义检索            │   │
│  └──────────────────────────────────────────┘   │
│                    │                             │
│                    ▼                             │
│  融合与调度                                      │
│  ┌──────────────────────────────────────────┐   │
│  │  RRF (Reciprocal Rank Fusion) — 混合融合   │   │
│  │  MMR (Maximal Marginal Relevance) — 去重   │   │
│  │  时间衰减 — 旧记忆权重自动降低              │   │
│  └──────────────────────────────────────────┘   │
│                    │                             │
│                    ▼                             │
│  对外接口                                        │
│  ┌──────────────────────────────────────────┐   │
│  │  memory_search  — 语义 + 关键词混合检索     │   │
│  │  memory_get     — 按路径/ID精确获取        │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  记忆固化机制（仿生学睡眠模型）                    │
│  ┌──────────────────────────────────────────┐   │
│  │  深度睡眠：大批量整合、压缩日记              │   │
│  │  浅度睡眠：更新 memory.md 中的长期画像       │   │
│  │  REM (快速眼动)：跨文件联想、建立新连接      │   │
│  │  机制：不是每条记忆都升级到长期记忆，        │   │
│  │        通过阈值和条件筛选                    │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

**核心特点：**

- **四层递进记忆**：`CLAUDE.md`（项目规则）→ `MEMORY.md`（用户长期画像）→ Session Memory（当前会话）→ AutoDream（后台记忆反思整合）
- **文件即真相**：所有记忆都是 Markdown 文件，用户可以直接打开查看、编辑、用 git 做版本管理
- **索引是派生品**：SQLite FTS5 和 LanceDB 只是文件内容的索引加速层，即使索引损坏，文件本身依然可读
- **睡眠 = 记忆固化**：模仿人脑的睡眠周期（深度睡眠/浅度睡眠/REM），在"睡眠"时异步整合记忆——合并碎片、更新长期画像、建立跨文件的语义连接
- **不做结构化抽取**：不依赖 LLM 把对话事实"抽取"成结构化数据存入数据库，而是直接在文件中追加文本——这避免了抽取环节的信息损失和错误传播

> **Claude Code 的逻辑**：与其让 LLM 把对话"翻译"成结构化记录（可能翻错），不如直接让 LLM 在需要时读原始文件。这就是文件派的"真相最小加工"原则。

### 6.4 OpenClaw 记忆设计：文件派 + 可插拔

OpenClaw 由 Peter Steinberger 创建并开源。我理解它是在文件派基础上，做了一次更系统的“基础设施化”改造：

```
┌─────────────────────────────────────────────────┐
│                   OpenClaw                        │
│                                                  │
│  双层记忆模型（默认）                              │
│  ┌──────────────────────────────────────────┐   │
│  │  MEMORY.md   — 永久性记忆（关键事实/经验）   │   │
│  │  每日日志     — 当前会话详细记录             │   │
│  │  自动刷写     — 70%上下文触发记忆整合        │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  可插拔 Context Engine                           │
│  ┌──────────────────────────────────────────┐   │
│  │  memory_search — 语义检索召回               │   │
│  │  可替换后端    — 默认文件，可接入任何引擎     │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  Gateway 架构（多入口 → 单控制平面）              │
│  ┌──────────────────────────────────────────┐   │
│  │  Web │ CLI │ API │ MCP │ IDE Plugin       │   │
│  │          ↓ 全部经过 Gateway ↓              │   │
│  │       统一的记忆读写 + 权限控制              │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

**OpenClaw 的独特设计：**

- **70%阈值自动刷写**：当上下文窗口使用率达到70%时，自动触发记忆整合——将当前会话的关键信息写回MEMORY.md，释放上下文空间。这是一个工程化的"防溢出"机制
- **Gateway 统一入口**：所有入口（Web/CLI/API/MCP/IDE插件）全部经过Gateway进入同一控制平面，记忆的读写权限在Gateway层统一管理——解决了多入口场景下的记忆一致性问题
- **可插拔 Context Engine**：默认走文件搜索，但可以通过插件接入 Mem0、Zep、向量数据库等任何记忆后端——OpenClaw不绑定记忆实现

### 6.5 Hermes 记忆设计：技能闭环 + 可插拔记忆

Hermes 由 [Nous Research](https://github.com/nousresearch/hermes-agent) 于 2025 年 7 月创建、2026 年 2 月公开发布，是2026年增长最快的开源 Agent 项目（MIT 协议，~90K Stars）。我把它的核心理念概括为：**“让 Agent 越用越聪明。”**——通过技能闭环将每次成功经验固化为可跨平台复用的资产。

**Hermes 记忆架构全景：**

```
┌──────────────────────────────────────────────────────────────────┐
│                        Hermes Agent                               │
│                                                                   │
│  记忆写入流程（每次对话后异步执行）                                    │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  用户消息 + Agent 回复                                        │ │
│  │       │                                                      │ │
│  │       ▼                                                      │ │
│  │  [后台线程] ───→ 记忆提取 ───→ 写入各记忆层                     │ │
│  │       │                                                      │ │
│  │       ├──→ MEMORY.md（Agent 对环境的认知）                      │ │
│  │       ├──→ USER.md   （Agent 对用户的画像）                    │ │
│  │       └──→ SKILL.md  （可复用的任务解决方案）                    │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                                                                   │
│  记忆检索流程（每次对话前预取）                                      │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  用户新消息                                                    │ │
│  │       │                                                      │ │
│  │       ▼                                                      │ │
│  │  [预取层] ───→ 从各记忆层检索相关记忆                           │ │
│  │       │                                                      │ │
│  │       ├──→ Session Context（当前会话，直接保留）                │ │
│  │       ├──→ MEMORY.md + USER.md（冻结进 System Prompt）         │ │
│  │       ├──→ SQLite FTS5（关键词全文检索历史）                    │ │
│  │       ├──→ 可插拔 Provider（Mem0 / Honcho / 自定义）            │ │
│  │       └──→ Skill 检索（匹配当前任务的已有技能）                   │ │
│  │                                                                 │
│  │  检索结果 → 去重 + 摘要压缩 → 注入上下文                         │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                                                                   │
│  五层记忆存储                                                      │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  第一层 Session Context        │  短期，当前对话窗口             │ │
│  │  第二层 MEMORY.md + USER.md    │  事实层，冻结入 System Prompt  │ │
│  │  第三层 SKILL.md               │  技能层，agentskills.io 标准  │ │
│  │  第四层 SQLite FTS5            │  全文索引层，历史对话关键词检索  │ │
│  │  第五层 Honcho 用户建模         │  行为层，异步心理建模 + 12身份层│ │
│  └─────────────────────────────────────────────────────────────┘ │
│                                                                   │
│  可插拔记忆 Provider（v0.7.0+）                                    │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  Provider ABC 接口                                             │ │
│  │  ├── 内置默认（文件后端 MEMORY.md + USER.md）                   │ │
│  │  ├── Honcho（实体行为建模，profile-scoped）                    │ │
│  │  ├── Mem0（语义搜索 + Reranking + 服务端事实提取）              │ │
│  │  ├── 向量数据库（Qdrant / Pinecone 等）                       │ │
│  │  └── 自定义（实现 Provider ABC 即可）                          │ │
│  │                                                                 │
│  │  可靠性保障：                                                   │ │
│  │  ├── Circuit Breaker：连续5次失败 → 暂停2分钟 → 自动重试        │ │
│  │  ├── 所有 API 调用在后台线程中非阻塞执行                          │ │
│  │  └── Memory Flush 状态持久化，Gateway 重启不丢失                │ │
│  └─────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

**核心特点：**

- **五层递进记忆**：Session（当前）→ Facts（MEMORY.md + USER.md）→ Skills（SKILL.md）→ FTS5（全文检索）→ Honcho（行为建模）。每层解决不同时间尺度和抽象层次的问题
- **双目标记忆**：`memory`（Agent 对工作环境的认知）和 `user`（Agent 对用户的画像）——两者独立管理，独立检索
- **零延迟预取**：每轮对话前，缓存的上轮记忆已就绪——不是等用户发消息后再检索，而是前一回合结束后异步预取
- **容量精控**：记忆容量被刻意控制在"足够小以保持速度，足够大以保持价值"的平衡点上
- **跨平台 Skills**：技能遵循 [agentskills.io](https://agentskills.io) 标准（YAML frontmatter + Markdown），可在 Claude Code、Cursor、GitHub Copilot、VS Code 等 11+ 工具间复用

Hermes 的记忆体系不是平面堆叠，而是按**时间尺度 + 抽象层次**两个维度组织的：

| 层级 | 存储 | 持久性 | 时间尺度 | 核心职责 |
|---|---|---|---|---|
| **Session Context** | Transformer 上下文窗口 | ❌ 会话级 | 秒~分钟 | 当前对话的即时推理状态 |
| **Persistent Facts** | MEMORY.md + USER.md | ✅ 永久 | 天~年 | Agent 对世界的认知 + 对用户的画像 |
| **Procedural Skills** | SKILL.md | ✅ 永久 | 周~年 | 可复用的任务解决方案 |
| **FTS5 Full-Text** | SQLite FTS5 索引 | ✅ 永久 | 全部历史 | 对话历史的关键词级快速检索 |
| **Honcho User Model** | Honcho 后端 | ✅ 永久 | 跨会话 | 用户行为模式 + 12 层身份建模 |

从三巨头看记忆设计的演进趋势，三条路线正在**殊途同归地收敛**：

```
Claude Code ──→ 加了 auto memory 和 channels ──┐
                                                 ├──→ 都在建设
OpenClaw ────→ 推进 Context Engine 可插拔化 ────┤     "有记忆、有技能、
                                                 │      有入口、有权限、
Hermes ──────→ 技能闭环 + 可插拔 Provider ──────┘      有持久状态"的个人AI系统
```

## 7. Harness Engineering：我眼里的 Agent Memory 工程方法论

> 如果说 Claude Code、OpenClaw、Hermes 更像是在回答“产品怎么实现记忆”，那 Harness 回答的就是另一个问题——**"记忆系统怎么设计才正确？"**

"Harness" 是2026年由 lencx 等人系统化提出的一套 Agent 工程概念体系，Martin Fowler 将其定义为 "Coding Agent 的信任建设模型"。它不是某个具体产品，而是一套设计方法论——用来回答：当我们构建一个值得信任的 Agent 时，记忆子系统应该满足哪些工程约束？


![](images\6.png)

### 7.1 Harness 的七层架构与记忆的位置

Harness 将 Agent 系统分为七层，记忆占据第二层——但它不是孤立组件，而是贯穿全部七层的"血液"：

```
┌──────────────────────────────────────────────────────────────┐
│              Harness Engineering：七层架构中的记忆              │
│                                                               │
│  第一层 角色与规则                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  Memory 提供"我是谁"的答案                                 │ │
│  │  → 用户画像决定 Agent 的角色边界                            │ │
│  │  → 项目上下文（CLAUDE.md）定义 Agent 能做什么、不能做什么     │ │
│  │  → 约束（constraint）型记忆直接参与行为边界定义             │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  第二层 记忆系统 ★── 本教程的核心讨论对象                       │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  Write → Manage → Read 三链路闭环                          │ │
│  │  存储/索引/检索/整合/遗忘——记忆系统的全部力学集中在这一层     │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  第三层 上下文加载                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  Memory 检索结果在此层进入上下文窗口                         │ │
│  │  → Token 预算分配：记忆 vs System Prompt vs 工具定义        │ │
│  │  → 不是"全部塞进去"，而是基于任务约束做取舍                  │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  第四层 稳定执行                                                │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  工具调用成功/失败 → 程序性记忆（什么方法 work/不 work）      │ │
│  │  错误与修复路径 → 经验积累。关键：不是所有细节都该记住         │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  第五层 有效循环                                                │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  Agent Loop 的每次迭代都可能触发记忆读写                     │ │
│  │  → 发现新约束/事实 → 即时写入 Working Memory                │ │
│  │  → Loop 受阻 → 检索类似经验                                │ │
│  │  → Loop 收口 → 关键发现写回长期记忆                         │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  第六层 评分可观测                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  记忆质量在这一层被检验                                     │ │
│  │  → 检索召回率/精确率？                                     │ │
│  │  → 记忆是否真的影响了决策？（Memory Utilization Rate）       │ │
│  │  → "僵尸记忆"占比？（从未被检索过的记忆）                    │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  第七层 中断恢复                                                │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  记忆是中断恢复的"锚点"                                     │ │
│  │  → Session State + Working Memory 恢复上下文               │ │
│  │  → 恢复后优先加载：当前任务状态 → 相关长期记忆 → 最新事件    │ │
│  └─────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```
> **核心认知：在 Harness 的视角下，Memory 不是第二层的孤岛，而是贯穿七层的"血液"。每一层都在消费记忆或生产记忆——设计时不能只看第二层。**

### 7.2 记忆治理：Harness 的核心命题

Harness 体系对记忆的最大贡献，是把**治理（Governance）**提到了和存储同等的高度。传统记忆讨论经常停留在“存哪、怎么查”，而 Harness 追问的是：

**五个治理命题：**

| 命题 | 核心问题 | Harness 的回答 |
|---|---|---|
| **所有权** | 这段记忆属于谁？用户、项目，还是 Agent？ | 三级所有权模型：User-scope → Project-scope → Agent-scope，每级有独立的 CRUD 权限 |
| **生命周期** | 记忆从创建到销毁经过哪些阶段？ | Working → Candidate → Consolidated → Archived → Purged，每个阶段有明确的升级/降级条件 |
| **可信度** | Agent 凭什么相信这段记忆？ | Provenance Chain（来源链）：从原始对话 → LLM 提取 → 写入 Ledger → 多次验证，每一步都记录 |
| **冲突仲裁** | 两个来源给出矛盾信息怎么办？ | 不自动合并，保留冲突并建模为"情境依赖"；由更高层策略（或用户）仲裁 |
| **遗忘权** | 用户能不能让 Agent "忘掉"某件事？ | 硬删除（从 Ledger 中移除）+ 软遗忘（Tombstone 标记，保留审计记录但不再检索） |

> 说白了，Provenance Chain 就是给记忆上“出生证明”。Agent 不能只说“我记得”，还得能说“我为什么这么记得”。

**治理的技术基座——Provenance Chain（来源链）：**

```
用户说："我不喜欢 ORM"
      │
      ▼
  [原始对话记录] ─── source_id: conv_20260429_001, turn: 12
      │
      ▼
  [LLM 提取] ─── extractor: claude-opus-4-6, confidence: 0.92, type: assertion
      │
      ▼
  [写入 Ledger] ─── transaction_id: tx_8a3f, timestamp: 2026-04-29T10:23:00Z
      │
      ▼
  [行为验证] ─── 连续 3 次手写 SQL，强化置信度 → confidence: 0.97
      │
      ▼
  [整合到画像] ─── MEMORY.md: "用户偏好原生 SQL 而非 ORM（高置信，行为验证）"
```

> **没有 Provenance Chain，Agent 就无法判断自己的"信念"有多可靠。这是 Harness 区分"记忆系统"和"随便存点东西"的关键分界线。**

Harness 把每条记忆视为一个状态机，而不是一行静态数据：

```
                    ┌─────────────┐
                    │   Working   │  ← 刚提取，还在验证中
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
       被后续证据强化   被后续证据削弱   超时未再出现
              │            │            │
              ↓            ↓            ↓
      ┌────────────┐  ┌──────────┐  ┌──────────┐
      │ Candidate  │  │  Disputed│  │ Transient│
      │  (候选)    │  │  (存疑)   │  │ (暂存)   │
      └─────┬──────┘  └────┬─────┘  └────┬─────┘
            │              │              │
       达到置信阈值    用户/新证据仲裁    超时清理
            │              │              │
            ↓              ↓              ↓
      ┌────────────┐  ┌──────────┐  ~~~~~~~~~~~~~
      │Consolidated│  │  Resolved│   (删除)
      │  (已整合)   │  │ (已裁决)  │
      └─────┬──────┘  └────┬─────┘
            │              │
      长期未引用/     ↓      ↓
      主动遗忘    ┌──────────────┐
            │     │   Archived   │  ← 保留但不再主动检索
            ↓     │   (已归档)    │
      ┌──────────┐ └──────┬───────┘
      │ Purged   │        │
      │ (已清除)  │   可恢复 ←→ 不可恢复
      └──────────┘
```

### 7.3 Harness 记忆的三个设计模式

从工程实践角度，我会把 Harness 里的记忆设计模式归纳成三类：

**模式一：分层注入（Layered Injection）**

我的理解是：不同记忆不要一锅炖。项目规则、用户画像、经验片段、日志摘要，各自该在什么时候出现，就在什么时候出现。

不同层级的记忆在不同时机、以不同方式进入上下文：

```
启动时注入（每次会话只加载一次）：
  ├── CLAUDE.md（项目规则，最高优先级，不可压缩）
  ├── MEMORY.md（用户画像，高优先级）
  └── 最近 N 天的每日日志摘要

按需检索（在推理过程中动态触发）：
  ├── memory_search(query) → 语义 + 关键词混合检索
  ├── memory_get(path)    → 精确获取特定记忆
  └── 检索结果经 Reranker + MMR 去重后注入

后台持续整合（会话间或空闲时）：
  ├── 日记压缩：合并碎片化记录为结构化摘要
  ├── 画像更新：从近期交互中提取新的偏好/事实
  └── 跨文件连接：建立不同日记间的语义关联
```

**模式二：预算感知检索（Budget-Aware Retrieval）**

这就像出门只带一个背包：什么都想塞，最后只会拉链爆炸。Context Window 也是一样。

记忆检索不是一个独立动作，而是一个在 Token 预算约束下的决策问题：

```
检索决策流程：
1. 当前上下文还剩多少 Token？（预算查询）
2. 当前任务需要什么类型的记忆？（约束分析）
         │
    ┌────┼────┐
    ↓    ↓    ↓
  偏好  规范  经验
  记忆  记忆  记忆
    │    │    │
    └────┼────┘
         ↓
3. 按优先级排序，在预算内尽可能多地加载
4. 预算不足时，优先加载高优先级、高置信、高时效的记忆
5. 被裁剪的记忆保留引用（memory_id），Agent 可在需要时二次检索
```

**模式三：经验闭环（Experience Loop）**

我觉得这一点最像“升级打怪”：每次任务成功或失败，都应该变成下一次可复用的经验，而不是让 Agent 每天都从新手村开始。

每次 Agent 执行任务都是一次"学习机会"：

```
任务开始
    │
    ▼
检索相关经验（程序性记忆）
    │
    ▼
执行任务，记录关键决策点
    │
    ▼
任务结束，评估结果
    │
    ├── 成功 → 提取成功模式 → 写入技能池（Skill Pool）
    │         ├── 触发条件是什么？
    │         ├── 具体步骤是什么？
    │         └── 什么情况下这个模式不适用？
    │
    └── 失败 → 提取失败原因 → 写入经验库
              ├── 根因是什么？
              ├── 修复路径是什么？
              └── 如何避免同类错误？
```

> **这三个模式共同构成了 Harness 记忆设计的工程内核：分层注入解决"什么时候给"、预算感知解决"给多少"、经验闭环解决"怎么越用越好"。**

### 7.4 上下文预算管理：记忆的“算账”问题

Harness 提醒了我一个很容易被忽略的问题：**记忆加载不是免费的，它在和 System Prompt、工具定义、Few-shot 示例竞争同一个有限的 Token 窗口。**

所以我会把 Token 预算显式分配到六个区域：

| 预算区域 | 内容 | 典型占比 | 特点 |
|---|---|---|---|
| 系统指令 | Agent 的角色定义、行为约束 | 15-20% | 固定，不可压缩 |
| 记忆加载 | MEMORY.md + 检索到的相关记忆 | 15-25% | 动态，按任务约束加载 |
| 工具定义 | 可用工具的 schema 和说明 | 10-15% | 固定，按需加载 |
| Few-shot 示例 | 过往成功案例的参考 | 5-10% | 动态，按任务类型选择 |
| 近期对话 | 当前会话的历史消息 | 20-30% | 滚动窗口，超限压缩 |
| RAG 知识 | 从外部知识库检索的文档 | 5-15% | 按需检索 |

**预算感知检索流程：**

```
检索决策流程：
1. 当前上下文还剩多少 Token？（预算查询）
2. 当前任务需要什么类型的记忆？（约束分析）
         │
    ┌────┼────┐
    ↓    ↓    ↓
  偏好  规范  经验
  记忆  记忆  记忆
    │    │    │
    └────┼────┘
         ↓
3. 按优先级排序，在预算内尽可能多地加载
4. 预算不足时，优先加载高优先级 × 高置信 × 高时效的记忆
5. 被裁剪的记忆保留引用（memory_id），Agent 可在需要时二次检索
```

> **Token 预算是硬约束。设计记忆系统时如果不做预算管理，再好的检索算法也会被"塞不下"的现实打败。**

### 7.5 Harness 对记忆工程实践的启示

我更愿意把 Harness 看成一份“设计备忘录”，而不是某个产品——当你在实战中选择或构建记忆系统时，它提供了一组不因产品更迭而过期的校验清单：

**如果我自己要做一个记忆系统，我会先拿这份清单自检：**

1. **Ledger 存在吗？** 你的记忆系统有没有 append-only 的权威记录？删除/更新操作是否可追溯？
2. **Provenance 可查吗？** 每条记忆能否回溯到原始对话？Agent 能否回答"我为什么相信这个"？
3. **治理闭环完整吗？** Write → Manage → Read 三条链路是否完整？Manage 链路有没有做冲突检测和衰减？
4. **预算感知了吗？** 记忆加载是否考虑了 Token 预算约束？有没有优先级排序机制？
5. **有状态机吗？** 记忆是"存了就存了"，还是经过了 Working → Candidate → Consolidated 的生命周期筛选？
6. **遗忘机制就绪吗？** 系统是否只增不减？有没有 Tombstone、衰减、归档、清除的逐级遗忘路径？
7. **可观测吗？** 你能回答"当前有多少条记忆"、"僵尸记忆占比"、"检索命中率"吗？

> **Harness 的核心主张：Memory 的难点从来不只在容量，更在治理。一个没有治理能力的记忆系统，最终一定会被自己的历史拖垮。**

## 8. 最后，我的总结

1. **Memory 不是存储，是闭环系统**：Raw Ledger（权威）→ Views（可用）→ Policy（控制）→ Commit（回写）→ Provenance（可回放）。三者缺一，系统要么不可治理，要么不可用，要么不可持续迭代。

2. **System 2（记忆层）是必要的**：记忆能力与 LLM 通用 Agent 能力是"相对正交"的，外置化带来可插拔、可迁移、可归因的工程好处。

3. **记忆的上限由三类瓶颈决定**：接口带宽（注入容量）、检索与聚合误差（views近似）、policy 的可学习性与可控性。其中 policy 往往是最被低估的瓶颈。

4. **时序是架构的结构维度，不是 metadata**：bi-temporal + time-sliced recall 是区分"曾经为真"与"当前为真"的硬约束，不能交给 LLM 自洽猜测。

5. **程序性记忆（技能）是长期能力增长的关键载体**：不只是记"是什么"，更要记"怎么做"。ProcMEM 证明可执行、可验证、可治理的技能池可以带来显著的性能收益。

6. **记忆治理与存储同等重要**：Harness 体系揭示了记忆的五个治理命题——所有权、生命周期、可信度（Provenance Chain）、冲突仲裁、遗忘权。记忆不只是"存哪、怎么查"，更是"谁拥有、凭什么信、怎么修正、何时遗忘。"

7. **记忆是有生命周期的状态机**：每条记忆从 Working → Candidate → Consolidated → Archived → Purged，经过明确的升级/降级条件。记忆不是"要么存要么删"的二元选择，而是一个可回溯、可恢复的灰度生命历程。

8. **统一范式尚未形成，这是窗口期**：2026年的记忆系统百花齐放，不论是文件派、数据库派还是模型派，都有各自的空间。

9. **选择性遗忘（Unlearning）是该还的技术债**：只增不减的记忆系统最终一定会被自己的历史拖垮。
