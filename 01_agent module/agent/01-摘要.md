# 摘要：Agent 是 Harness 制度内的行动主体

在 Harness Engineering 的语境中，Agent 不是简单的大模型，也不是可以无限授权的自动化脚本。更准确的理解是：Agent 是运行在 Harness 制度内的行动主体。它围绕目标持续理解状态、读取上下文、调用工具、观察反馈、修正策略，并在必要时与其他 Agent 协作。

这个定义把 Agent 从“会说话的模型”推进到“受控行动者”。它既承认模型的推理和生成能力，也强调这种能力必须被放在工具、权限、状态、记忆、验证和生命周期治理之中。

37 篇文章和 4 个 codewiki 项目显示，Agent 职责不应被无限扩大。Agent 可以负责理解、计划、行动、解释反馈、总结、协作和有限学习；但目标函数、权限边界、验证标准、长期记忆治理、系统安全和最终价值判断，不能完全交给 Agent 自己。否则越强的 Agent，越可能放大错误目标、错误记忆或错误权限。

成熟 Agent 设计的关键，不是让 Agent “更自主”，而是让它在清晰边界内越来越能干。OpenHarness 用 QueryEngine、ToolRegistry、Skill、Memory 和 Coordinator 定义 Agent 的行动条件；HiClaw 用 Manager-Worker、Matrix、MinIO 和 Higress 区分调度、执行、通信和凭证；Hermes 用 SessionDB、ACP Adapter、MCP、可选 Skill 和 cron 支撑常驻 Agent；DeerFlow 用 LangGraph Lead Agent、中间件、沙盒和 IM 渠道管理平台级职责。

这些项目共同说明：Agent 的职责体系不是抽象伦理问题，而是具体工程边界设计。哪些事由 Agent 判断，哪些事由 Harness 强制，哪些事必须人类介入，是 Agent 系统能否可靠的核心。
