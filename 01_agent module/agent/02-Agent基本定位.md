# Agent 的基本定位

如果只看模型，Agent 很容易被理解为“会调用工具的 LLM”。这个定义太薄。它解释不了为什么同一个模型放在不同系统里，表现会差异巨大。

更准确地说，Agent 是在 Harness 提供的运行环境中，围绕目标持续决策、调用工具、观察结果、更新状态并推进任务的行动单元。

这个定义里，“目标”很关键。没有目标，模型只是对话系统；有了目标，它才需要计划、行动和验证。一个问答模型回答“如何修复 bug”时，它只是在生成建议；一个工程 Agent 接收“修复这个 bug”时，它必须定位、修改、测试、解释和交付。

“行动”也很关键。Agent 的行动并不直接来自模型，而是通过工具实现。文件系统、Shell、浏览器、API、消息平台、沙盒、子 Agent，都是它行动的载体。工具让 Agent 进入现实，也让风险进入现实。

“反馈”则决定 Agent 是否真的形成闭环。Agent 不是一次性输出答案，而是在工具结果、测试失败、用户反馈、日志异常或评测结果返回后继续判断：是否成功、是否偏航、是否要重试、是否要停下。

因此，Agent 的本质不是“更会说话的模型”，而是“被 Harness 赋予行动能力，并被 Harness 约束和纠偏的行动主体”。这个定位决定了后续所有职责划分。
