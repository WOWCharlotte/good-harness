# 工程 Agent：从“会写代码”到“会交付变更”

工程 Agent 的职责不是“写代码”，而是交付可验证变更。

第一，它要先读代码。真实项目中的模式、边界、依赖、测试入口和历史包袱，往往比模型的通用知识更重要。没有读过代码就提方案，是工程 Agent 的常见失败。

第二，它要控制修改范围。好的 Agent 不会借一个小 bug 重构半个系统。它会围绕目标做最小变更，并解释为什么这个改动足够。

第三，它要运行验证。格式化、类型检查、单测、集成测试、总验证脚本，都是工程交付的一部分。验证失败时，Agent 要继续诊断，而不是把失败留给用户。

第四，它要识别技术债。Agent 会放大代码库中的模式。坏模式如果大量存在，Agent 可能误以为那是规范。因此，Agent 既要学习现有模式，也要判断哪些模式可能是历史遗留。

第五，它要留下可审查说明。人类需要知道改了什么、为什么改、验证如何、风险在哪里。

所以，工程 Agent 的成熟标志不是生成代码速度，而是能否在真实仓库中稳定完成“理解-修改-验证-说明”的闭环。
